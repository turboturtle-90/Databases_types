### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему? 

`1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.`

С учетом необходимой целостности, четкой структуры и необходимости выполнения сложных запросов для аналитики лучшим решением будет `реляционная база данных SQL`.
 

`1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.`

Не являясь специалистом ни в лендинге, ни в CRM, на основе анализа терминов и подходов сделал следующие выводы:
- для лендингов, с учетом тесной работы с сайтом и необходимости быстроты лучшим решением кажется `noSQL`, вероятно, `документо-ориентированная`.
- для CRM, где необходимо хранить связи и, вероятно, нужна аналитика, `SQL` будет лучшим решением.


`1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.`

Если речь идет об отражении структуры компании и структуре базы, которая должна быть максимально понятной, лучшим решением будет `NoSQL база иерархического типа`.


`1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.`

Лучшим решением будет база `графового типа` для хорошей проработки связей и логистики.

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

Для транзакций обязательно соблюдение принципов ACID (атомарность, согласованность, изоляция, долговечность). 

Следующие шаги должны произойти: 
1. Пользователь отправляет запрос на пополнение. Проверяется существование номера телефона, наличие средств на счету для их перевода. 
2. Создание записи о транзакции со статусом «В обработке» (Pending). Это необходимо для аудита и возможности восстановления в случае сбоя.
3. Блокировка ресурсов: СУБД временно блокирует строку с балансом пользователя, чтобы предотвратить одновременное изменение этого счета другими процессами (например, списанием за тариф в ту же секунду).
4. Проведение платежа, взаимодействие с внешним банковским шлюзом. Система получает подтверждение, что деньги успешно списаны со счета отправителя.
5. Обновление баланса: В рамках одной атомарной транзакции значение баланса в БД увеличивается на сумму пополнения, а статус транзакции меняется на «Завершено»
6. Фиксация и уведомление. СУБД сохраняет изменения на диск (Commit). После подтверждения записи блокировка снимается, и пользователю отправляется уведомление о зачислении средств.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL. 

1. Гарантия целостности (соблюдение ACID). SQL-базы строго следуют принципам атомарности и согласованности. 
2. Стандартизированный язык SQL.
3. Строгая схема данных. Это обеспечивает высокое качество данных на входе
4. Сложные связи и JOIN. Удобно делать различные аналитические запросы.
5. Нормализация. Данные хранятся без дублирования.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

MapReduce это модель распределенных вычисления типовая для всех HPC систем. ЗАдача проходит декомпозицию, куски выделяются нодам для обработки -mapping. Ноды могут иметь свою RAM или современные класстеры имеют распределенную память. Когда идет счет или обработка, ноды могут взаимодействовать друг с другом если задача того требует. Когда результаты готовы, делается обратная композиция -Reduce.

Основными критериями при таком HPC будут свойственные PA-системам, т.е. доступность и устойчисвость при декомпозиции. Также важны масштабируемость, поскольку разные задачи требуют разных ресурсов, но база должна быть универсальной. Также важна in-memory обработка данных.

По всем признакам наиболее подходит `noSQL база key-value с in-memory поддержкой`.

Из личного опыта HPC (вычислительная газодинамика Ansys CFX/Fluent), могу добавить, что часто решатели не привязаны к коммерческим БД, а все процессы с данными собственно решателем и управляются, с задействованием, конечно, инструментов MPI. Так делают потому, что решатели часто имеют специфические особенности (явные, неявные, механизмы сходимости и т.п.). Каждое действие с данными завязывать на базу вызовет еще большую латентность на класстере. К тому же, объем данных которые собственно сохраняются как результат расчета в десятки раз меньше данных, которые висят в оператвной памяти при решении. 
Но и без отсутствия БД принципы работы с данными в оперативной памяти (ну или по сути это своего рода in-memory БД) как указаны выше - масшитабирование, доступность, способность надежно декомпозировать задачу.
